## What to Look For

### Premise & Approach
- **Problem framing**: Is the stated problem the real problem, or a symptom of something deeper?
- **Fundamental assumptions**: What assumptions does this plan rest on? Are they valid?
- **Alternative approaches**: Is there a simpler, more effective, or fundamentally different way to achieve the same goal?
- **Scope appropriateness**: Is this plan solving the right problem at the right level? Is it too narrow or too broad?
- **Cost vs. benefit**: Is the proposed effort justified by the expected outcome, or is there a higher-leverage path?

### Completeness
- **Missing requirements**: Are all features fully specified?
- **Incomplete tasks**: Do tasks have enough detail to implement?
- **Missing dependencies**: Are task dependencies properly identified?
- **Missing edge cases**: Are error conditions and edge cases addressed?
- **Missing tests**: Is testing strategy adequate?

### Correctness
- **Technical feasibility**: Is the proposed approach technically sound?
- **Architecture issues**: Are there design flaws or anti-patterns?
- **Security concerns**: Are there potential vulnerabilities?
- **Performance issues**: Will the approach scale appropriately?
- **Integration issues**: Will the changes work with existing code?

### Clarity
- **Ambiguous requirements**: Are any requirements unclear or open to interpretation?
- **Vague tasks**: Do tasks clearly specify what needs to be done?
- **Missing context**: Is there sufficient background for implementers?
- **Conflicting statements**: Are there contradictions in the plan?

### Implementation Quality
- **Over-engineering**: Is the solution more complex than necessary?
- **Under-engineering**: Is the solution too simplistic for the problem?
- **Code organization**: Is the proposed file structure sensible?
- **Naming consistency**: Are names clear and consistent throughout?

## Output Format
Return ONLY a valid JSON array with no additional text. Each element must have exactly these fields:
- "title": string (5-10 word summary of the issue)
- "desc": string (text explaining the issue and recommended fix)
- "importance": "high" | "medium" | "low"
  - high = critical gaps, security issues, architectural flaws, blockers
  - medium = missing details, unclear requirements, potential issues
  - low = minor improvements, style suggestions, nice-to-haves
- "reference": string (which section of the plan this affects, e.g., "## Requirements" or "### Task 3")
- "type": "addition" | "modification" | "deletion" | "clarification"
  - addition = something is missing and should be added
  - modification = something exists but needs to be changed
  - deletion = something should be removed (unnecessary, harmful)
  - clarification = something is unclear and needs better explanation
